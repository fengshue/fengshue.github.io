React发展历史

传统页面 : 从服务器获取数据，显示到浏览器上，用户输入数据传入服务器的时候，随着数据量的增大，越来越难以维护。
Facebook认为MVC无法满足他们的扩展需求，由于他们非常巨大的代码库和庞大的组织，使得MVC很快变得非常复杂，每当需要添加一项新的功能或特性时，系统的复杂度就成级数增长，致使代码变得脆弱和不可预测，结果导致他们的MVC正在土崩瓦解。认为MVC不适合大规模应用，当系统中有很多的模型和相应的视图时，其复杂度就会迅速扩大，非常难以理解和调试。（MVC：http://www.ruanyifeng.com/blog/2015/02/mvcmvp_mvvm.html）

Model（模型）:
是应用程序中用于处理应用程序数据逻辑的部分。 通常模型对象负责在数据库中存取数据。

View（视图）:
是应用程序中处理数据显示的部分。 通常视图是依据模型数据创建的。

Controller（控制器）:
是应用程序中处理用户交互的部分。 通常控制器负责从视图读取数据，控制用户输入，并向模型发送数据。

理论知识

主要原理和特点

效率高

虚拟DOM：传统web页面操作DOM一般是直接操作，代价很大，涉及重绘重排非常耗性能，而React 为了尽可能减少对DOM的操作，提供了一种不同的而又强大的方式来更新DOM，代替直接操作DOM,就是Virtual dom, 一个轻量级虚拟的dom。更新virtual dom时不保证马上影响真实dom，react会等到事件循环结束，然后利用diff算法，通过当前新的dom与之前做比较，计算出最小的步骤更新真实的dom。

天生组件化

Components组件：在dom树上的节点被称为元素，在这里不同称为virtual dom，整体称为component， 众多的virtual dom的节点就是一个完整的抽象的组件。

清晰

单项数据流
从父节点传递到子节点，因为组件是简单而且易于把握的，他们只需从父节点获取props渲染即可。

State 和 Render
state属性包含定义组件所需要的一些数据，当数据发生变化时，将会调用Render重现渲染，
React 把组件看成是一个状态机（State Machines）。通过与用户的交互，实现不同状态，然后渲染 UI，让用户界面和数据保持一致。

2. 两个对象
元素对象（canvas元素）和上下文对象（通过getContext(‘2d’)方法获取到的CanvasRenderingContext对象）。
元素对象相当于画布，上下文对象相当于画笔，接下来所有操作是基于上下文对象的。

3. 线段
ctx.moveTo(x,y);移动到x，y坐标点

ctx.closePath();闭合当前路径和回到起始点的区别
ctx.fill();填充
ctx.fillStyle=’green’;设置填充颜色
* 重点：fill 和 stroke方法都是作用在当前的所有子路径
完成一条路径后要重新开始另一条路径时，必须使用beginPath()方法（开始子路径的一个新的集合）。

特殊点：strokeRect和fillRect自带beginPath属性。

异步加载js

一、js同步加载

  1. js加载的缺点：加载工具方法没必要阻塞文档，使得js加载会影响页面效率，一旦网速不好，那么挣个网站将等待js加载而不进行后续渲染等工作。
  2. js加载不能满足的需求：有些工具方法需要按需加载，用到时再加载，不用时不加载。
  3. 页面解析：加载一行，解析一行，遇到标签，开启一个新线程，异步加载css。

    二、js异步加载

  4. javascript异步加载的三种方案：
    1） defer异步加载，但要等到dom文档全部解析完才会被执行（只有IE能用。执行时不阻塞页面）

2） async异步加载，加载完就执行（async只能加载外部脚本，不能将js写在script标签里，w3c标准。

执行时不阻塞页面）
3） 创建script，插入到DOM中，onload，兼容，加载完毕后callBack。( * 按需加载 * ）

js加载时间线

js时间线：

  1. 创建Document对象，开始解析页面。解析HTML元素和他们的文本内容后，添加Element对象和Text节点到文档中，这个阶段document.readyState = ‘loading’。
  2. 遇到link外部css，创建线程加载，并继续解析文档。
  3. 遇到script外部js并且没有设置defer或async，浏览器加载并阻塞，等待js加载完成并执行该脚本后，继续解析文档。
  4. 遇到script外部js并且有设置defer或async，浏览器常见线程加载，并继续解析文档。对于async属性，脚本加载完成后立即执行（异步禁止使用document.write）
  5. 遇到img等，先正常解析dom结构，然后浏览器开始加载src，并继续解析文档。
  6. 当文档解析完成，document。readyState=’interactive’（document.onreadystatechange）
  7. 文档解析完成后，所有设置有defer的脚本会按照顺序执行（注：与async不同，但同样禁止document.write）
  8. document对象触发DOMContentloaded事件，这也标志着程序执行从同步脚本执行阶段，转换为事件驱动阶段（document.addEventListener）
  9. 当所有async的脚本加载完成并执行后、img等加载完成后，docuemnt.readyState = ‘complete’，window对象触发load事件。
  10. 从此，以一部相应方式处理用户输入，网络事件等。
    docuemnt.write()能清除文档流，故异步加载禁止使用document.write()

iphone状态栏兼容问题

在app里，当客户端去掉原生的titlebar时，前端的h5页面在iphone里就会出现页面从带有时间，信号，电量的那个“状态栏”里开始展示的问题（安卓不会）。本来：

viewport-fit=cover表示：全屏模式，但是对于iphone来说，这个设置无效，都会出现上述问题。于是，探究出两套解决方案。先普及一下：

导航栏的高度应该是44px。顶部状态栏（Darger Area）的高度：对于iphoneX系列（X、Xs、Xs Max、XR）来说应该是44px，对于除此之外的iphone（8plus、8、7plus、7、6splus、6s、SE等）来说应该20px。

清楚了这一点之后，我们来看两套解决方式：
1.通过js判断。区分出iphonx系列，再区分出非iphoneX系列的iphone，之后分别添加不用类名的样式。

2.通过css-media媒体查询。区分出是否是iphone，为iphone添加同意的类名样式，在用媒体查询为iphoneX系列添加特殊样式。
